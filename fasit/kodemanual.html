Fase 1 — Oppsett STEG 01

Installer Python og VS Code

Verktøyene du skriver og kjører koden i.

Gå til python.org og last ned Python. Under installasjonen er det én ting som er kritisk:

⚠️
Ikke glem dette

Huk av "Add Python to PATH" under installasjonen. PATH er en liste datamaskinen bruker til å finne programmer du skriver i terminalen. Uten dette vil python-kommandoen ikke fungere.

Last så ned VS Code fra code.visualstudio.com. Etter installasjon, åpne Extensions (Ctrl+Shift+X) og installer Python (av Microsoft) og Jinja.

🧠
Hvorfor VS Code og ikke Notepad?

VS Code forstår Python — den markerer feil mens du skriver, foreslår kode du kan trykke Tab for å fullføre, og har en innebygd terminal. Det er som å skrive en tekst i Word vs. Notepad. Du kan bruke Notepad, men du velger det vekk raskt.

Test at Python er installert riktig. Åpne terminalen i VS Code med Ctrl + ` og skriv:

Terminal 
python --version
# Forventer: Python 3.11.x eller lignende
Hva skjer her?

Du ber terminalen kjøre programmet python med argumentet --version. Programmet skriver ut versjonen og avslutter. Ser du en versjon → alt OK. Ser du "not recognized" → PATH er ikke satt riktig, installer på nytt.

✍️
Din oppgave

Installer Python og VS Code. Åpne terminalen og kjør python --version. Skriv ned hvilken versjon du fikk før du går videre.

Fase 1 — Oppsett STEG 02

Mappestruktur og virtuelt miljø

Bygg grunnmuren — ryddig fra dag én.

Lag en ny mappe et sted på datamaskinen, kalt laeringsplattform. Åpne den i VS Code via File → Open Folder. Lag deretter denne strukturen inne i mappen:

laeringsplattform/ ├── templates/ # Flask leter alltid her etter HTML │ ├── base.html # Felles ramme alle sider arver │ ├── login.html │ ├── register.html │ ├── index.html │ ├── section.html │ ├── topic.html │ ├── profile.html │ └── admin/ │ ├── panel.html │ └── setup.html ├── static/ # Flask leter her etter CSS, bilder, JS │ └── style.css └── app.py # Alt Python-kode bor her
🧠
Hvorfor akkurat disse mappenavnene?

Flask er et "konvensjonsbasert" rammeverk. Det betyr at det har hardkodet inn forventninger: det leter i templates/ når du kaller render_template(), og i static/ når du bruker url_for('static', ...). Du kan endre dette, men det er ingen grunn til det — følg konvensjonen og ting bare fungerer.

Virtuelt miljø

Åpne terminalen i VS Code og kjør disse kommandoene én om gangen:

Terminal 
# Lag et virtuelt miljø kalt "venv"
python -m venv venv

# Aktiver det (Windows)
venv\Scripts\activate

# Aktiver det (Mac / Linux)
source venv/bin/activate

# Legg merke til at "(venv)" dukker opp i terminalen — bra!
# Installer alle pakker vi trenger
pip install flask flask-sqlalchemy flask-login werkzeug markdown
Linje for linje
  • python -m venv venv — -m venv betyr "kjør den innebygde Python-modulen kalt venv". Det lager en mappe kalt venv/ med en isolert Python-installasjon.
  • venv\Scripts\activate — Bytter "aktiv Python" i denne terminalen til den i venv/. Fra nå av er all pip install isolert til dette prosjektet.
  • pip install ... — pip er Pythons pakkebehandler. Den laster ned og installerer pakker fra nettet (fra pypi.org).
🧠
Hvorfor virtuelt miljø — kan jeg ikke bare pip installere globalt?

Jo, men tenk deg at du har to prosjekter: ett trenger Flask 2.0, et annet trenger Flask 3.0. Globalt kan du bare ha én versjon installert — de to prosjektene ville krasje. Et virtuelt miljø gir hvert prosjekt sin egen boble med sine egne pakker i sine egne versjoner. Det er standarden i profesjonell Python-utvikling.

PakkeHva den gjør

flask-loginHåndterer innlogging, sesjoner og "hvem er innlogget nå"

markdownKonverterer Markdown-tekst til HTML
⚠️
Husk dette hver gang du åpner VS Code

Hvert nytt terminalvindu starter uten venv aktivert. Kjør alltid venv\Scripts\activate før du jobber med prosjektet. Ser du (venv) i prompten → aktivert. Ser du det ikke → ikke aktivert.

Fase 2 — Flask-grunnlag STEG 03

Den første Flask-appen

Minimal fungerende server — og hva hver eneste linje faktisk gjør.

Åpne app.py og skriv inn dette. Les forklaringene nøye — dette er skjelettet resten av appen henger på.

app.py Python 
from flask import Flask, render_template

app = Flask(__name__)
app.config['SECRET_KEY'] = 'bytt-meg-til-noe-hemmelig-123'

@app.route('/')
def index():
    return render_template('index.html')

if __name__ == '__main__':
    app.run(debug=True)
Linje for linje — hva skjer egentlig?
  • from flask import Flask, render_template — Hent to ting fra Flask-pakken: Flask (klassen som representerer appen) og render_template (funksjonen som leser HTML-filer og sender dem til nettleseren).
  • app = Flask(__name__) — Lag en Flask-app. __name__ er en innebygd Python-variabel som inneholder navnet på den gjeldende modulen. Flask bruker dette til å finne prosjektets rotmappe — og dermed der templates/ og static/ befinner seg.
  • app.config['SECRET_KEY'] = '...' — En hemmelig streng Flask bruker til å signere session-cookies kryptografisk. Uten den kan ikke Flask-Login fungere. Bytt den til noe tilfeldig og langt.
  • @app.route('/') — En dekorator. Den registrerer funksjonen under som "handler" for URL-en /. Når nettleseren besøker http://localhost:5000/, kalles denne funksjonen.
  • def index(): — Vanlig Python-funksjon. Navnet index er bare et navn — du kan kalle den hva som helst.
  • return render_template('index.html') — Les filen templates/index.html, prosesser den som en Jinja2-mal, og returner HTML-strengen. Flask sender denne tilbake til nettleseren.
  • if __name__ == '__main__': — Denne blokken kjøres bare hvis du kjører python app.py direkte. Hvis app.py importeres av en annen fil, kjøres ikke blokken. Standard Python-mønster.
  • app.run(debug=True) — Start Flask sin innebygde webserver. debug=True gjør to ting: serveren restarter automatisk når du lagrer endringer, og feilmeldinger vises i nettleseren med full stack trace.
🧠
Hva er en dekorator egentlig?

En dekorator er Python-syntaks for å pakke inn en funksjon. @app.route('/') er det samme som å skrive index = app.route('/')(index) — det registrerer funksjonen i Flasks interne URL-kart. Tenk på det som å si til Flask: "Legg til denne funksjonen som handler for / i oppslagstabellen din."

Lag en minimal templates/index.html for å teste:

templates/index.html HTML 
<!DOCTYPE html>
<html><body>
  <h1>Flask fungerer! 🎉</h1>
</body></html>

Kjør appen og åpne nettleseren:

Terminal 
python app.py
# Gå til: http://localhost:5000
✍️
Prøv selv

Endre teksten i index.html mens serveren kjører. Refresh nettleseren — ser du endringen umiddelbart? Det er debug=True i aksjon. Prøv også å gi route-funksjonen et annet navn — hva skjer?

Fase 2 — Flask-grunnlag STEG 04

base.html og Jinja2-arv

Skriv navbar én gang. Få den på alle sider automatisk.

🧠
Hvorfor arv i HTML-maler?

Uten arv: Du skriver navbaren i alle 9 HTML-filer. Vil du endre én lenke → 9 steder å endre. Med arv: base.html inneholder navbaren og all felles HTML. Alle andre filer sier bare "jeg bygger på base" og fyller inn selve innholdet. Endre navbaren ett sted → oppdateres overalt automatisk.

templates/base.html HTML + Jinja2 
<!DOCTYPE html>
<html lang="no">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>{% block title %}Læringsplattform{% endblock %}</title>
  <link rel="stylesheet"
        href="{{ url_for('static', filename='style.css') }}">
</head>
<body>

  <nav>
    <a href="/">🏠 Hjem</a>

    {% if current_user.is_authenticated %}
      <a href="/profil">Min profil</a>

      {% if current_user.is_admin %}
        <a href="/admin">Admin</a>
      {% endif %}

      <span>{{ current_user.username }}</span>
      <a href="/logg-ut">Logg ut</a>
    {% else %}
      <a href="/login">Logg inn</a>
    {% endif %}
  </nav>

  <main>
    {% with messages = get_flashed_messages(with_categories=true) %}
      {% for category, message in messages %}
        <p class="alert {{ category }}">{{ message }}</p>
      {% endfor %}
    {% endwith %}

    {% block content %}{% endblock %}
  </main>

</body>
</html>
Jinja2-syntaks forklart
  • {% block title %}Læringsplattform{% endblock %} — Definer et navngitt hull kalt "title". Teksten mellom er standardverdien — brukes hvis en undermal ikke overskriver blokken.
  • {{ url_for('static', filename='style.css') }} — Doble krøllparenteser = skriv ut en verdi. url_for er en Flask-hjelpefunksjon som genererer korrekt URL til filer i static/. Alltid bruk denne istedenfor hardkodede stier.
  • {% if current_user.is_authenticated %} — Enkle krøllparenteser med % = kontrollstruktur (if, for, block osv.). current_user er et spesialobjekt fra Flask-Login, alltid tilgjengelig i alle maler.
  • {{ current_user.username }} — Skriv ut brukernavnet til den innloggede brukeren.
  • {% block content %}{% endblock %} — Det viktigste hullet — her settes alt sideinnhold inn av undermaler.
  • get_flashed_messages(with_categories=true) — Henter midlertidige meldinger lagret med flash() i Python. De vises kun én gang — neste gang siden lastes er de borte.

Slik bruker en side base.html:

templates/index.html — oppdatert versjon Jinja2 
{% extends "base.html" %}

{% block title %}Forsiden{% endblock %}

{% block content %}
  <h1>Hei, {{ current_user.username }}! 👋</h1>
{% endblock %}
Hva skjer under panseret?

Når Flask kaller render_template('index.html'), leser Jinja2 filen, ser {% extends "base.html" %}, laster base.html, og setter innholdet fra {% block content %} inn i hullet i base. Resultatet er én komplett HTML-side som sendes til nettleseren.

✍️
Prøv selv

Oppdater index.html til å arve fra base.html som vist over. Hva skjer i nettleseren? Du vil se en feilmelding om current_user — det er fordi Flask-Login ikke er satt opp ennå. Det fikser vi i steg 6. For nå: fjern {{ current_user.username }} og se at arven fungerer.

Fase 3 — Database STEG 05

Databasemodeller med SQLAlchemy

Beskriv databasen som Python-klasser — ikke SQL-spørringer.

🧠
Hva er et ORM og hvorfor bruker vi det?

ORM = Object-Relational Mapper. Uten ORM måtte du skrive "INSERT INTO users (username, password) VALUES (?, ?)" og håndtere SQL-feil manuelt. Med SQLAlchemy skriver du Python-klasser, og SQLAlchemy oversetter til SQL bak kulissene. Fordelen: du tenker i Python-objekter, ikke tabeller og kolonner. Koden blir tryggere og mer lesbar.

Legg til dette øverst i app.py, rett etter den eksisterende koden:

app.py — imports og konfigurasjon Python 
from flask import (Flask, render_template, redirect,
                    url_for, request, flash, jsonify)
from flask_sqlalchemy import SQLAlchemy
from flask_login import (LoginManager, UserMixin,
                              login_user, logout_user,
                              login_required, current_user)
from werkzeug.security import generate_password_hash, check_password_hash
from functools import wraps
import markdown as md

app = Flask(__name__)
app.config['SECRET_KEY']            = 'bytt-meg-til-noe-hemmelig-123'
app.config['SQLALCHEMY_DATABASE_URI'] = 'sqlite:///database.db'

db = SQLAlchemy(app)
Hva er nytt her?
  • SQLALCHEMY_DATABASE_URI = 'sqlite:///database.db' — Fortell SQLAlchemy hvor databasen er. sqlite:/// betyr SQLite (filbasert database). De tre skråstrekene etterpå betyr "relativ til prosjektmappen". Filen havner i instance/database.db.
  • db = SQLAlchemy(app) — Kobler databasen til Flask-appen. Nå kan du bruke db overalt i prosjektet.
Modellene
app.py — alle databasemodeller Python 
# ── BRUKER ──────────────────────────────────────────
class User(UserMixin, db.Model):
    id            = db.Column(db.Integer, primary_key=True)
    username      = db.Column(db.String(80),  unique=True,  nullable=False)
    password_hash = db.Column(db.String(200),                  nullable=False)
    is_admin      = db.Column(db.Boolean, default=False)

# ── SEKSJON ─────────────────────────────────────────
class Section(db.Model):
    id          = db.Column(db.Integer, primary_key=True)
    title       = db.Column(db.String(100), nullable=False)
    description = db.Column(db.Text)
    emoji       = db.Column(db.String(10))
    topics      = db.relationship('Topic', backref='section', lazy=True,
                                  cascade='all, delete-orphan')

# ── TEMA ────────────────────────────────────────────
class Topic(db.Model):
    id         = db.Column(db.Integer, primary_key=True)
    title      = db.Column(db.String(100), nullable=False)

    elements   = db.relationship('LearningElement', backref='topic', lazy=True,
                                  cascade='all, delete-orphan')

# ── LÆRINGSELEMENT ──────────────────────────────────

# ── ÅPNE SVAR ───────────────────────────────────────
class OpenAnswer(db.Model):
    id         = db.Column(db.Integer, primary_key=True)
    user_id    = db.Column(db.Integer, db.ForeignKey('user.id'),             nullable=False)
    element_id = db.Column(db.Integer, db.ForeignKey('learning_element.id'), nullable=False)
    answer     = db.Column(db.Text)

# ── TEMABESØK ───────────────────────────────────────
class TopicVisit(db.Model):
    id       = db.Column(db.Integer, primary_key=True)
    user_id  = db.Column(db.Integer, db.ForeignKey('user.id'),    nullable=False)
    topic_id = db.Column(db.Integer, db.ForeignKey('topic.id'),   nullable=False)
Nøkkelkonsepter
  • class User(UserMixin, db.Model) — Klassen arver fra to foreldre. db.Model gjør den til en databasetabell. UserMixin er fra Flask-Login — den legger til metoder som is_authenticated, is_active og get_id() som Flask-Login forventer å finne på bruker-objektet.
  • db.Column(db.Integer, primary_key=True) — En heltall-kolonne som er primærnøkkel. SQLite teller den automatisk opp (1, 2, 3...) for hver ny rad.
  • db.ForeignKey('section.id') — Pek på en annen tabell. 'section.id' betyr "id-kolonnen i section-tabellen". Dette er relasjonen mellom to tabeller.
  • db.relationship('Topic', backref='section', cascade='all, delete-orphan') — Definer en Python-relasjon. backref='section' betyr du kan skrive topic.section for å hente seksjonen. cascade='all, delete-orphan' betyr: slett alle temaer automatisk hvis seksjonen slettes.
🧠
Hvorfor lagre passord som "hash" og ikke klartekst?

En hash er en enveisfunksjon: "hemmelig123" → "$2b$12$xK...". Det er umulig å gå tilbake. Selv om noen stjeler databasen, ser de bare hash-verdier — ikke passordene. check_password_hash(lagret_hash, forsøk) hasher forsøket med samme algoritme og sjekker om de matcher — uten å "dekryptere" noe.

Fase 3 — Database STEG 06

Opprett databasen og admin-bruker

Kjør db.create_all() og se databasefilen dukke opp.

Legg til denne koden i app.py, etter alle modellene men før rutene:

app.py — initialiser DB Python 
with app.app_context():
    db.create_all()

    if not User.query.filter_by(username='admin').first():
        admin = User(
            username='admin',
            password_hash=generate_password_hash('admin123'),
            is_admin=True
        )
        db.session.add(admin)
        db.session.commit()
Hva skjer steg for steg?
  • with app.app_context(): — Flask bruker en "kontekst" for å vite hvilken app som er aktiv. Utenfor en HTTP-forespørsel (som under oppstart) finnes ingen kontekst automatisk. app_context() oppretter en manuelt.
  • db.create_all() — Går gjennom alle klasser som arver fra db.Model, og lager tilsvarende SQL-tabeller. Tabeller som allerede eksisterer røres ikke — trygt å kjøre hver gang appen starter.
  • User.query.filter_by(username='admin').first() — Søk etter en bruker med brukernavn "admin". .first() returnerer første treff eller None — brukes til å sjekke om admin allerede finnes.
  • db.session.add(admin) — Legg til objektet i den pågående transaksjonen (men skriv ikke til disk ennå).
  • db.session.commit() — Skriv alle endringer i transaksjonen til databasefilen permanent.
🧠
Hvorfor add() og så commit() — ikke bare save()?

SQLAlchemy bruker en transaksjon-modell. Du kan legge til mange objekter med add() uten at noe skrives til databasen. Først når du kaller commit() skrives alt på én gang. Fordelen: hvis ett objekt feiler, rulles hele gruppen tilbake — databasen blir aldri halvveis oppdatert.

Kjør nå python app.py. Du skal se at mappen instance/ dukker opp med en fil kalt database.db.

✍️
Utforsk databasen

Last ned DB Browser for SQLite (gratis, fra sqlitebrowser.org) og åpne instance/database.db. Du kan se tabellene og admin-raden som er opprettet. Dette er nyttig for å forstå hva som faktisk lagres.

Fase 4 — Autentisering STEG 07

Flask-Login — sesjoner og cookies

Hvordan appen husker deg mellom sidebesøk — og hvorfor det er nødvendig.

🧠
Problemet: HTTP er statsløst

Hver HTTP-forespørsel er uavhengig. Serveren har ingen innebygd hukommelse — etter at den svarte på "last forsiden" har den glemt at du i det hele tatt besøkte. Uten sesjoner måtte du logge inn på nytt for hver eneste side.

Løsningen: Sesjons-cookie. Etter innlogging lagrer Flask en kryptert cookie i nettleseren din. Nettleseren sender denne med på alle fremtidige forespørsler — som et ID-kort. Serveren leser ID-kortet og vet hvem du er.

Legg til dette i app.py, etter db = SQLAlchemy(app):

app.py — Flask-Login oppsett Python 
login_manager = LoginManager(app)
login_manager.login_view = 'login'

@login_manager.user_loader
def load_user(user_id):
    return User.query.get(int(user_id))
Hva skjer her?
  • LoginManager(app) — Kobler Flask-Login til appen. Fra nå av er current_user tilgjengelig i alle ruter og maler.
  • login_manager.login_view = 'login' — Hvis noen prøver å gå til en @login_required-beskyttet side uten å være innlogget, send dem til ruten kalt 'login'. Legg merke til at det er funksjonsnavnet, ikke URL-en.
  • @login_manager.user_loader — Registrer en "loader-funksjon". Hver gang Flask mottar en forespørsel med en session-cookie, kaller Flask-Login denne funksjonen med bruker-ID-en fra cookien. Vi henter brukeren fra databasen og returnerer den — Flask-Login setter den som current_user.
  • int(user_id) — Cookies lagrer alltid tekst (strenger). Vi konverterer til heltall fordi databasens ID er et heltall.

Beskytt nå forsiden. Legg @login_required på index-funksjonen:

app.py — beskyttet rute Python 
@app.route('/')
@login_required       # ← legg til denne
def index():
    sections = Section.query.all()
    return render_template('index.html', sections=sections)
To dekoratorer — rekkefølgen betyr noe

Dekoratorer kjøres nedenfra og opp. Flask sjekker først om URL-en matcher '/', deretter sjekker @login_required om brukeren er innlogget. Hvis ikke, redirectes brukeren til login_view. Rekkefølgen @app.route øverst og @login_required under er standard — alltid gjør det slik.

Fase 4 — Autentisering STEG 08

Login, registrering og utlogging

GET vs POST, skjemadata og passord-hashing i praksis.

🧠
Hvorfor trenger en route både GET og POST?

En login-side gjør to ting: GET = "vis meg skjemaet" (nettleseren besøker URL-en). POST = "behandle dataene jeg nettopp sendte inn" (bruker klikket Submit). Flask skiller disse ved hjelp av request.method. Alternativet er to separate URL-er for hvert skjema — mer rotete og vanskelig å lenke til.

app.py — login, register, logg-ut Python 
@app.route('/login', methods=['GET', 'POST'])
def login():
    if request.method == 'POST':
        username = request.form['username']
        password = request.form['password']

        user = User.query.filter_by(username=username).first()

        if user and check_password_hash(user.password_hash, password):
            login_user(user)
            return redirect(url_for('index'))
        else:
            flash('Feil brukernavn eller passord', 'error')

    return render_template('login.html')


@app.route('/register', methods=['GET', 'POST'])
def register():
    if request.method == 'POST':
        username = request.form['username']
        password = request.form['password']

        if User.query.filter_by(username=username).first():
            flash('Brukernavnet er allerede tatt', 'error')
            return redirect(url_for('register'))

        ny = User(
            username=username,
            password_hash=generate_password_hash(password),
            is_admin=False
        )
        db.session.add(ny)
        db.session.commit()
        login_user(ny)
        return redirect(url_for('index'))

    return render_template('register.html')


@app.route('/logg-ut')
@login_required
def logg_ut():
    logout_user()
    return redirect(url_for('login'))
Nøkkelpunkter
  • request.form['username'] — Henter verdien fra HTML-skjemafeltet med name="username". Navnet i HTML og Python må stemme overens.
  • user and check_password_hash(...) — To betingelser med and. Hvis user er None (brukernavn finnes ikke), evalueres aldri check_password_hash — Python short-circuits.
  • flash('melding', 'error') — Lagrer en engangsmelding. Den vises én gang i base.html via get_flashed_messages(), og forsvinner deretter.
  • url_for('index') — Genererer URL-en til ruten kalt index. Ikke skriv redirect('/') hardkodet — url_for er tryggere og fungerer selv om du endrer URL-en.
HTML — login.html
templates/login.html HTML + Jinja2 
{% extends "base.html" %}
{% block title %}Logg inn{% endblock %}
{% block content %}

Hva betyr "dynamisk URL"?

def topic(topic_id):
    tema = Topic.query.get_or_404(topic_id)

    # Registrer besøk første gang
    if not TopicVisit.query.filter_by(
            user_id=current_user.id, topic_id=topic_id).first():
        db.session.add(TopicVisit(user_id=current_user.id, topic_id=topic_id))

    # Hvilke elementer har brukeren allerede svart på?
    answered_ids = {
        p.element_id
        for p in Progress.query.filter_by(user_id=current_user.id).all()
    }
    open_answers = {
        a.element_id: a.answer
        for a in OpenAnswer.query.filter_by(user_id=current_user.id).all()
    }

    # Konverter markdown til HTML
    elements_rendered = []
    for el in tema.elements:
        rendered = el.content
        if el.type == 'markdown':
            rendered = md.markdown(el.content,
                                   extensions=['fenced_code', 'tables'])
        elements_rendered.append((el, rendered))

    return render_template('topic.html',
        topic=tema,
        elements_rendered=elements_rendered,
        answered_ids=answered_ids,
        open_answers=open_answers
    )
To Python-teknikker verdt å forstå
  • {p.element_id for p in Progress.query...} — Et set comprehension. Lager et Python-sett (som en liste, men uten duplikater og med rask oppslag). Brukes fordi vi trenger å sjekke if element.id in answered_ids mange ganger — sett gjør dette mye raskere enn en liste.
  • {a.element_id: a.answer for a in OpenAnswer.query...} — Et dict comprehension. Lager en ordbok der nøkkelen er element_id og verdien er svaret. I malen kan vi da skrive open_answers[element.id] for å hente svaret direkte.
  • elements_rendered.append((el, rendered)) — Legger til et tuppel (to verdier pakket sammen) til listen. I malen loopes over dem med {% for element, rendered in elements_rendered %}.

Fase 6 — Interaktivitet STEG 11

Quiz med fetch() — ingen sidereload

Asynkron kommunikasjon: JavaScript snakker med Flask i bakgrunnen.

🧠
Hva er problemet med vanlige skjema-submit for quiz?

Med et vanlig HTML-skjema lastes hele siden på nytt etter submit. Alle quizer returnerer til utgangs­posisjonen, scrollposisjonen nullstilles, og bruker­opplevelsen er dårlig. Med fetch() sendes svaret til serveren i bakgrunnen, vi mottar JSON tilbake, og oppdaterer bare de berørte knappene — ingen reload.

Flask-ruten som mottar quiz-svar:

app.py — quiz og åpent svar Python 
@app.route('/answer/quiz', methods=['POST'])
@login_required
def answer_quiz():
    data       = request.get_json()
    element_id = data['element_id']
    answer     = data['answer']

    element = LearningElement.query.get_or_404(element_id)

    if Progress.query.filter_by(
            user_id=current_user.id, element_id=element_id).first():
        return jsonify({'error': 'allerede besvart'}), 400

    correct = (answer == element.correct_answer)
    db.session.add(Progress(
        user_id=current_user.id,
        element_id=element_id,
        correct=correct
    ))
    db.session.commit()

    return jsonify({
        'correct': correct,
        'right_answer': element.correct_answer
    })


@app.route('/answer/open/<int:element_id>', methods=['POST'])
@login_required
def answer_open(element_id):
    element = LearningElement.query.get_or_404(element_id)
    if not OpenAnswer.query.filter_by(
            user_id=current_user.id, element_id=element_id).first():
        db.session.add(OpenAnswer(
            user_id=current_user.id,
            element_id=element_id,
            answer=request.form['answer']
        ))
        db.session.commit()
    return redirect(url_for('topic', topic_id=element.topic_id))
GET vs POST vs JSON
  • request.get_json() — Når JavaScript sender JSON (ikke et HTML-skjema), bruker du denne istedenfor request.form. Den parser JSON-teksten til en Python-ordbok.
  • jsonify({...}) — Konverter en Python-ordbok til en JSON-tekststreng og returner den som HTTP-svar med Content-Type: application/json.
  • return jsonify(...), 400 — Det andre tallet er HTTP-statuskoden. 400 = "Bad Request". JavaScript kan sjekke response.ok for å se om det gikk bra.
JavaScript i topic.html
templates/topic.html — script-tag JavaScript 
<script>
document.querySelectorAll('.quiz-btn').forEach(btn => {

  btn.addEventListener('click', async () => {

    // Finn quiz-blokken denne knappen tilhører
    const block     = btn.closest('.quiz-block')
    const elementId = block.dataset.elementId
    const answer    = btn.dataset.answer

    // Send JSON til Flask
    const response = await fetch('/answer/quiz', {
      method:  'POST',
      headers: { 'Content-Type': 'application/json' },
      body:    JSON.stringify({ element_id: elementId, answer: answer })
    })

    if (!response.ok) return  // allerede besvart

    const data = await response.json()

    // Farg alle knapper i denne quiz-blokken
    block.querySelectorAll('.quiz-btn').forEach(b => {
      b.disabled = true
      if (b.dataset.answer === data.right_answer) {
        b.classList.add('correct')
      } else if (b === btn && !data.correct) {
        b.classList.add('wrong')
      }
    })
  })
})
</script>
JavaScript-konsepter forklart
  • async () => { ... } — En asynkron pilfunksjon. async gjør at vi kan bruke await inni — det stopper funksjonen og venter på svar fra serveren uten å blokkere resten av siden.
  • btn.closest('.quiz-block') — Klatre opp i DOM-treet til første foreldrelement med klassen quiz-block. Nyttig når en knapp er nestet inne i en container.
  • block.dataset.elementId — Leser data-element-id-attributtet. Legg merke til at bindestrek konverteres til camelCase automatisk.
  • JSON.stringify({...}) — Konverter JavaScript-objekt til JSON-tekst for sending.
  • await response.json() — Vent på og parse JSON-svaret fra Flask til et JavaScript-objekt.
  • b.classList.add('correct') — Legg til en CSS-klasse dynamisk. I style.css kan du style .quiz-btn.correct til å bli grønn.
Fase 7 — Admin STEG 12

Admin-beskyttelse og CRUD

Lag din egen dekorator og håndter oppretting/sletting av innhold.

🧠
Hvorfor lage en egen dekorator istedenfor bare å sjekke inne i funksjonen?

Du kunne skrevet if not current_user.is_admin: abort(403) øverst i hver admin-funksjon. Men da er sjekken spredd ut over mange steder — og lett å glemme. En dekorator pakker inn logikken én gang og brukes som en merkelapp. Det er DRY — Don't Repeat Yourself.

app.py — admin_required og CRUD Python 
from flask import abort

# ── EGENDEFINERT DEKORATOR ──────────────────────────
def admin_required(f):
    @wraps(f)
    @login_required
    def decorated(*args, **kwargs):
        if not current_user.is_admin:
            abort(403)
        return f(*args, **kwargs)
    return decorated


# ── ADMIN PANEL ─────────────────────────────────────
@app.route('/admin')
@admin_required

    sections = Section.query.all()
    return render_template('admin/panel.html', sections=sections)

Dekoratoren under lupen
  • def admin_required(f): — Tar inn en funksjon f og returnerer en ny funksjon som pakker den inn.
  • @wraps(f) — Uten dette ville den innpakkede funksjonen miste sitt opprinnelige navn og docstring. @wraps bevarer dem — nødvendig for at Flask skal kunne identifisere ruter korrekt.
  • abort(403) — Flask-funksjon som avbryter forespørselen med HTTP-statuskoden 403 (Forbidden). Nettleseren vil vise en feilside.
  • return f(*args, **kwargs) — Kjør den opprinnelige funksjonen med alle argumentene den fikk. *args og **kwargs er Python sin måte å si "ta alle argumenter og send dem videre".

app.py — profil-rute Python 
@app.route('/profil')
@login_required
def profile():
    uid = current_user.id

    answered = Progress.query.filter_by(user_id=uid).count()
    correct  = Progress.query.filter_by(user_id=uid, correct=True).count()
    visited  = TopicVisit.query.filter_by(user_id=uid).count()

    # Fremgang per tema
    topic_stats = []
    for tema in Topic.query.all():
        total = len(tema.elements)
        if total == 0:
            continue
        done = Progress.query.filter_by(user_id=uid).join(
            LearningElement
        ).filter(LearningElement.topic_id == tema.id).count()
        topic_stats.append({
            'topic': tema,
            'done': done,
            'total': total,
            'percent': int(done / total * 100)
        })

    return render_template('profile.html',
        answered=answered, correct=correct,
        visited=visited, topic_stats=topic_stats
    )
JOIN i SQLAlchemy — koble to tabeller

CSS-konsepter som er verdt å forstå dypt
  • :root { --primary: #4f46e5; } — CSS custom properties (variabler). Endrer du fargen her, oppdateres den overalt i filen automatisk. Essensielt for vedlikehold.
  • position: sticky; top: 0; z-index: 100; — Navigasjonen forblir synlig mens du scroller. z-index: 100 sørger for at den legger seg over annet innhold.
  • grid-template-columns: repeat(auto-fill, minmax(230px, 1fr)) — Responsivt rutenett uten media queries. Lag flest mulig kolonner som er minst 230px brede. Smalere skjerm → færre kolonner, alt automatisk.
  • transition: transform .15s, box-shadow .15s — Animer flere egenskaper samtidig. Hover-effekten på kortene er glatt fordi begge egenskapene animeres.
  • box-shadow: 0 0 0 3px rgba(79,70,229,.15) — Focus-ring: x-offset=0, y-offset=0, blur=0, spread=3px. En synlig, tilgjengelig markering av aktivt skjemafelt.

Markdown-forhåndsvisning i admin

Live rendering av Markdown mens admin skriver.

Legg til dette i admin-panelet der admin oppretter Markdown-elementer:

templates/admin/panel.html — Markdown preview HTML + JS 
<!-- Last inn marked.js fra CDN -->
<script src="https://cdn.jsdelivr.net/npm/marked/marked.min.js"></script>

<textarea
  id="md-input"
  name="content"
  rows="8"
  placeholder="Skriv Markdown her..."></textarea>

<div id="md-preview" class="markdown-content">
  Forhåndsvisning vises her...
</div>

<script>
  const input   = document.getElementById('md-input')
  const preview = document.getElementById('md-preview')

  input.addEventListener('input', () => {
    preview.innerHTML = marked.parse(input.value)
  })
</script>
Hva skjer?
  • 'input'-hendelsen — Fyres av hver gang innholdet i textarea endres (ved hvert tastetrykk). Raskere enn 'change' som bare fyres når textarea mister fokus.
  • marked.parse(input.value) — Biblioteket marked.js konverterer Markdown til HTML direkte i nettleseren — ingen serverrunde nødvendig for forhåndsvisning.
  • preview.innerHTML = ... — Setter HTML-innholdet til preview-diven. Siden vi stoler på Markdown-utgangen (admin skriver det selv), er det trygt å sette innerHTML direkte.

Hvorfor bare POST for sletting — ikke DELETE?

HTML-skjemaer støtter bare GET og POST. HTTP-metoden DELETE finnes, men nettlesere sender den aldri fra et vanlig skjema. Den brukes av JavaScript og API-er. For enkle webapper med HTML-skjemaer bruker vi POST for alle skriveoperasjoner (opprette, endre, slette). URL-strukturen forteller hva som gjøres: /delete i URL-en gjør intensjonen tydelig.

Gull-tips for feilsøking

Flask debugger: Når debug=True, vises feilmeldinger direkte i nettleseren med komplett stack trace. Les den fra bunnen — den underste linjen peker oftest på din kode.
